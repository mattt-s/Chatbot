# 群组能力 PRD

## 1. 背景

当前系统的核心对象是 `panel`，每个 `panel` 对应一个独立角色窗口，并映射到一个独立的 provider target。现有能力适合"单角色单会话"，不支持"一个群组内包含多个 agent 角色、按 @ 定向触发、角色间协作"的场景。

本需求要在现有"角色列表 + 聊天窗口"的产品模型上扩展出群组能力，支持用户在一个群组中组织多个 agent，通过 `@角色名` 触发消息路由，并实现角色之间的自主协作——由指定的组长角色（Leader）负责任务编排和消息调度。

## 2. 目标

实现一个可用的群组协作模式，使用户可以：

- 创建群组，并在左侧列表中查看群组。
- 在群组内添加多个角色，每个角色绑定一个 agent。
- 指定一个角色为组长（Leader），负责任务编排与兜底路由。
- 在群组聊天中通过 `@角色名` 触发指定角色处理消息。
- 角色处理完成后以该角色身份回复，并通过末尾 `@` 将消息转发给其他角色或组长。
- 角色之间通过 `@` 实现自主协作——串行、并行、任务拆分均由角色自身决定。
- 管理群组内角色，包括查看、设置组长和移除；移除后历史消息保留。

## 3. 非目标

- 不做 App 层编排引擎；编排权交给角色自身（尤其是 Leader）。
- 不做复杂权限系统；默认单用户使用。
- 不做跨群组共享角色实例。
- 不做数据库迁移；当前阶段仍基于本地 JSON 持久化。
- 不做全文搜索、消息置顶、群公告等扩展能力。
- 不做角色回复内容的智能拆分（多个角色同时被 @ 时，各角色收到完整的指令文本）。

## 4. 用户角色

- 管理员用户：当前唯一登录用户，负责创建群组、添加角色、设置组长、发送消息、管理群角色。

## 5. 核心概念

- **群组**：一个对话容器，拥有自己的消息流和成员角色列表。
- **群角色**：群组内的一个 agent 代理，具备显示名称和绑定的 agentId。
- **组长（Leader）**：群组内被指定的核心角色，负责接收兜底消息并做出路由决策。每个群组有且仅有一个组长。
- **消息路由**：App 根据消息末尾的 `@角色名` 自动将消息转发给对应角色。角色回复无 `@` 时，自动兜底转发给组长。
- **消息队列**：当目标角色正在推理（忙碌）时，待发送的消息排队等待，推理完成后合并发送。
- **消息协议**：每个角色首次被调用时，注入群组成员信息和消息格式规则，使角色理解"这条消息来自谁"以及"通过末尾 @ 指定转发给谁"。

## 6. 用户故事

### 6.1 创建群组

作为用户，我希望在左侧列表中点击"新增群组"，输入群组名称后创建一个群组，以便组织多个角色参与同一对话。

验收标准：

- 点击"新增群组"后弹出创建群组弹窗。
- 必填字段仅包含群组名称。
- 创建成功后，群组出现在左侧列表中。
- 新群组可被选中并进入聊天页面。

### 6.2 群组内添加角色

作为用户，我希望在群组页面点击"添加角色"，输入角色名称并选择一个 agent，把角色加入当前群组。

验收标准：

- 群组聊天页头部存在"添加角色"按钮。
- 弹窗标题为"添加角色"。
- 表单包含 `角色名称` 与 `选择 Agent`。
- 新创建角色仅出现在当前群组内，不出现在左侧角色列表顶层。

### 6.3 设置组长

作为用户，我希望在角色管理中将某个角色指定为组长，使其承担任务编排和兜底路由的职责。

验收标准：

- 角色管理弹窗中，每个角色旁有"设为组长"按钮。
- 当前组长旁显示红色「组长」标签。
- 设置新组长时自动取消旧组长的标记。
- 每个群组有且仅有一个组长。
- 未指定组长时，无 `@` 的角色回复不做兜底转发。

### 6.4 群组内角色列表

作为用户，我希望在群组详情中看到群内角色列表及组长标识，以便确认群组成员和职责分工。

验收标准：

- 左侧群组项可展开或展示其角色成员。
- 组长角色旁显示红色「组长」标签。
- 群组选中态与普通角色页有明显区分。
- 群组成员与顶层角色列表在视觉层级上可区分。

### 6.5 @ 触发消息处理

作为用户，我希望在群组消息里通过 `@角色1`、`@角色2` 指定角色处理消息，而不是所有角色都自动响应。

验收标准：

- 群组内角色在未被 `@` 时不主动生成回复。
- 一条用户消息可同时 `@` 多个角色。
- 被 `@` 的角色均可产生独立回复。
- 未被 `@` 的角色不会收到本轮处理任务。

### 6.5.1 @ 输入辅助

作为用户，我希望在群组输入框有便捷的 @ 角色方式。

验收标准：

- 在群组聊天输入框中输入 `@` 字符时，弹出角色候选列表。
- 输入框旁提供 **@ 按钮**，点击后直接弹出角色候选列表。
- 候选列表展示角色名 + emoji/头像，组长角色带「组长」标签。
- 候选列表中不包含当前用户自己。
- 候选列表只展示摘要，不展示树状展开结构。
- 角色名采用严格匹配规则。
- 点击候选项后，输入框自动插入 `@角色名 `，并保持可继续输入状态。

### 6.6 角色间自主协作

作为用户，我希望角色（尤其是组长）能够通过 @ 将任务委派给其他角色，其他角色完成后通过 @ 回传结果，实现角色间的自主协作。

验收标准：

- 角色回复末尾可包含 `@其他角色名`，系统自动将消息转发给被 @ 的角色。
- 一条角色回复可同时 `@` 多个角色，系统并行转发。
- 被转发的角色收到的消息中标明来源（来自哪个角色）。
- 角色回复无 `@` 时，系统自动将消息兜底转发给组长。
- 组长回复无 `@` 时，不再转发（视为最终输出）。
- 目标角色正在推理时，待转发的消息自动排队，推理完成后合并发送。
- 所有角色的回复（包括角色间的转发）均在群组聊天中可见。

### 6.7 消息气泡中的 @ 展示

作为用户，我希望消息气泡清晰地展示发送者身份和转发目标。

验收标准：

- 每条消息气泡左上角或顶部展示发送者角色名 + emoji/头像。
- 消息末尾的 `@角色名` 不在气泡正文中展示。
- 被 `@` 的角色名字和头像/emoji 展示在气泡右下角，与正文分离。
- 正文中间自然出现的 `@角色名`（非末尾）保留在正文中，不做剥离。

### 6.8 管理群组角色

作为用户，我希望点击"角色管理"查看群内角色并移除无效角色。

验收标准：

- 群组聊天页头部存在"角色管理"按钮。
- 点击后弹出角色管理窗口。
- 窗口中展示现有群角色列表，组长旁有红色「组长」标签。
- 每个角色旁有"设为组长"按钮和"移除"按钮。
- 移除后，该角色不再出现在群角色列表，也不再参与后续路由。
- 历史消息保留，不做回溯删除。
- 组长被移除后，群组暂无组长，无 `@` 的角色回复不做兜底转发。

## 7. 主要页面与交互

### 7.1 左侧列表

- 保留现有顶层"角色列表"结构。
- 顶层列表新增两类项目：普通角色、群组。
- 群组项可展示成员摘要或展开成员。
- 组长角色旁显示红色「组长」标签。

### 7.2 群组聊天页

- 头部展示群组名称，而不是某个单独角色名称。
- 头部新增按钮：`添加角色`、`角色管理`。
- 消息区允许出现来自多个群角色的回复消息。
- 每条消息气泡显示发送者角色标签。
- 消息末尾 `@` 在气泡右下角展示被 @ 角色的头像/emoji + 名字。
- 当有角色正在推理时，显示"XX 正在思考..."状态提示。
- 输入框不禁用——用户随时可以发消息，目标角色忙时消息自动排队。
- 输入框旁提供 @ 快捷按钮。

### 7.3 创建群组弹窗

- 输入：群组名称。
- 输出：创建群组实体，并进入群组页面。

### 7.4 添加角色弹窗

- 输入：角色名称、绑定 Agent。
- 输出：创建群组成员。

### 7.5 角色管理弹窗

- 列出群角色，组长角色标注红色「组长」标签。
- 支持"设为组长"操作。
- 支持"移除"操作。
- 明示"移除只影响后续消息，历史消息保留"。

## 8. 数据与状态要求

### 8.1 群组

群组复用现有 `panel` 实体，通过 `kind = "group"` 区分。至少需要以下属性：

- `id`
- `userId`
- `title`
- `kind`（`"direct"` | `"group"`）
- `createdAt`
- `updatedAt`

### 8.2 群角色

至少需要以下属性：

- `id`
- `panelId`（所属群组 panel 的 id）
- `agentId`
- `title`
- `emoji`（可选）
- `isLeader`（是否为组长）
- `enabled`（逻辑删除标记）
- `createdAt`
- `updatedAt`

### 8.3 群消息

消息需要区分：

- 发送者类型：用户 / 群角色 / 系统
- 发送者身份：用户本人或某个 `groupRoleId`
- 发送者名字快照：`senderLabel`（角色被删除后仍可展示）
- 末尾 @ 信息：`mentionedGroupRoleIds`（供前端气泡右下角展示）

## 9. 业务规则

### 9.1 消息路由

- 用户消息中 `@角色名` → 转发给被 @ 的角色。
- 角色回复末尾 `@角色名` → 转发给被 @ 的角色。
- 角色回复末尾无 `@` 且群组有组长 → 兜底转发给组长。
- 组长回复末尾无 `@` → 不转发，视为最终输出。
- 无组长的群组中，角色回复无 `@` → 不转发。
- 用户消息未 `@` 任何角色 → 前端提示"请 @ 一个角色"，不触发路由。
- `@角色名` 采用严格匹配，仅解析消息末尾的 `@`（最后 3 行）。

### 9.2 消息队列

- 目标角色正在推理时，待转发消息排入该角色的等待队列。
- 角色推理完成后，队列中的消息合并为一条发送。
- 队列深度有上限，超出时丢弃并提示。

### 9.3 角色管理

- 同名角色不允许在同一群组内重复创建。
- 不同群组内允许存在同名角色。
- 移除群角色不影响历史消息展示（发送者名字快照保留）。
- 群组删除策略不在本期范围；默认可先不做。
- 一条消息允许同时 `@` 多个角色。
- 角色不能 `@` 自己。
- 每个群组有且仅有一个组长。

### 9.4 消息协议

- 角色首次被调用时，注入群组成员信息、消息规则和回复格式示例。
- 组长额外注入「组长职责」说明。
- 成员列表中标注谁是组长。
- 每条发给角色的消息标明来源：`[来自 用户]` 或 `[来自 角色名]`。
- 角色只能看到发给自己的消息，无法直接看到其他成员之间的对话。

## 10. 边界情况

- 用户发送消息但未 `@` 任何角色：
  - 前端提示"请至少 @ 一个角色"，不触发路由。
- 用户 `@` 了不存在的角色：
  - 前端或服务端返回明确错误提示。
- 一条消息中重复 `@` 同一角色：
  - 去重，只触发一次。
- 输入框单独输入 `@` 时：
  - 弹出角色候选列表。
- 角色被移除后再收到旧 run 的异步回流：
  - 按历史 run 正常入库展示，但新消息不应再路由给该角色。
- 组长被移除：
  - 群组暂无组长，无 `@` 的角色回复不做兜底转发，直到用户指定新组长。
- 角色回复中正文里自然提到 `@角色名`（非末尾）：
  - 不触发路由，保留在正文中正常展示。
- 消息转发频率过高（可能的循环）：
  - 每个群组有每分钟最大转发次数限制，超限时暂停转发并提示。
- 角色不遵守消息协议（忘记写末尾 @）：
  - 兜底转发给组长，由组长根据语义判断下一步。

## 11. 成功指标

- 用户可稳定创建群组、添加角色、设置组长、发送 `@` 消息并收到对应角色回复。
- 组长能通过 @ 将任务委派给其他角色，其他角色完成后结果回传给组长。
- 角色回复无 @ 时，消息自动兜底到组长。
- 群角色移除后，后续消息不再触发该角色。
- 群组能力上线后，不回归现有单角色 panel 流程。

## 12. 发布建议

建议分两期：

### Phase 1

- 群组创建（复用 panel，`kind = "group"`）
- 群组内添加角色
- 设置组长
- `@角色` 消息路由（含末尾 @ 解析、兜底给组长）
- 消息队列（目标忙时排队，空闲后合并发送）
- 消息协议（入群提示词注入、来源标记）
- 角色管理（设为组长、移除）
- 安全阀（频率限制、队列深度限制）
- UI 完整可用（群组聊天页、@ 候选、气泡右下角 @ 展示、组长标签）

### Phase 2

- 群组展开/折叠交互优化
- 群组维度的统计、搜索、排序
- 消息协议的 few-shot 优化（提高角色遵守率）
- 群组会话持久化恢复（App 重启后恢复进行中的路由状态）

## 13. 已确认决策

- 群组复用 `panel` 实体，`kind = "group"`，不新增独立的群组容器。
- Gateway 侧零改动，每个角色有独立的 Gateway session。
- App 只做消息路由和排队，不做编排；编排权交给角色自身。
- 角色间通信通过 App 中转——App 伪装成"user"向角色 session 发送消息。
- `@角色名` 仅解析消息末尾（最后 3 行），正文中的 `@` 不触发路由。
- 前端 @ 展示规则与后端路由规则一致：只剥离末尾 @ 行，正文中的 `@` 保留。
- 消息队列和路由状态在 App 内存中管理，不持久化（Phase 1）。
- 所有角色的回复都在群组聊天中可见，不区分"对内"和"对外"。
- 输入框不禁用——用户随时可以发消息，目标忙时自动排队。
