# 群组能力测试文档

## 1. 测试目标

验证群组能力在当前项目架构下的正确性、稳定性和兼容性，覆盖：

- UI 交互正确性
- API 与持久化正确性
- `@` 路由逻辑正确性
- SSE/回流展示正确性
- 对现有单角色 panel 能力无回归

## 2. 测试范围

本次测试覆盖：

- 群组创建
- 群组列表展示
- 群组内添加角色
- 群组角色管理
- 群组消息发送与 `@` 路由
- 群角色回复展示
- 历史消息保留

不覆盖：

- 多用户并发访问
- 大规模性能压测
- provider 外部系统真实故障恢复演练

## 3. 测试环境

- Node.js 20
- Next.js 15.5
- 本地 JSON 存储：`storage/app-data.json`
- 本地 provider / customchat 插件联调环境

## 4. 测试类型

### 4.1 单元测试

建议覆盖：

- `@角色名` 解析函数
- 路由命中与去重逻辑
- 群角色增删改查的 store 层函数
- 群组视图组装函数
- 群消息 sender/target 映射函数

### 4.2 集成测试

建议覆盖：

- 群组创建 API
- 群组内添加角色 API
- 群角色移除 API
- 群消息发送 API
- provider 回流后消息入库与 SSE 推送

### 4.3 手工测试

建议覆盖主要前端交互和联调流程。

## 5. 详细测试用例

## 5.1 群组创建

### CASE-GRP-001 创建群组成功

前置条件：

- 用户已登录。

步骤：

1. 点击“新增群组”。
2. 输入合法群组名称。
3. 点击确认创建。

预期：

- 创建成功。
- 左侧列表出现新群组。
- 群组可被选中进入详情页。
- 存储中出现新的 group 记录。

### CASE-GRP-002 群组名称为空

步骤：

1. 打开创建群组弹窗。
2. 不输入名称直接提交。

预期：

- 前端禁用提交或显示校验错误。
- 不创建任何记录。

### CASE-GRP-003 重复群组名

步骤：

1. 创建群组 A。
2. 再次创建同名群组 A。

预期：

- 依据最终规则决定是否允许。
- 若不允许，应返回明确错误提示。

## 5.2 群组内添加角色

### CASE-GRP-ROLE-001 添加角色成功

前置条件：

- 已存在群组 A。
- Agent 列表可用。

步骤：

1. 进入群组 A。
2. 点击“添加角色”。
3. 输入角色名称。
4. 选择 Agent。
5. 提交。

预期：

- 创建成功。
- 新角色显示在群组 A 下。
- 新角色不显示在左侧顶层角色列表。
- 存储中出现 group role 记录。

### CASE-GRP-ROLE-002 角色名称为空

预期：

- 不允许提交。
- 无新增记录。

### CASE-GRP-ROLE-003 未选择 Agent

预期：

- 不允许提交。
- 无新增记录。

### CASE-GRP-ROLE-004 同群组内重复角色名

步骤：

1. 在群组 A 中创建角色 1。
2. 再创建同名角色 1。

预期：

- 返回冲突提示。
- 不重复创建。

### CASE-GRP-ROLE-005 不同群组允许同名角色

步骤：

1. 在群组 A 创建角色 1。
2. 在群组 B 创建角色 1。

预期：

- 创建成功。
- 两个角色分别归属于各自群组。

## 5.3 群组消息发送与 @ 路由

### CASE-ROUTE-001 单角色命中

步骤：

1. 在群组中发送 `@角色1 请处理`。

预期：

- 仅角色1被触发。
- 角色2不触发。
- UI 中出现角色1回复。

### CASE-ROUTE-002 多角色命中

步骤：

1. 发送 `@角色1 @角色2 请确认`。

预期：

- 角色1、角色2均被触发。
- 两者均可产生独立回复。

### CASE-ROUTE-003 输入单独 @ 时显示候选

步骤：

1. 进入群组聊天页。
2. 在输入框中输入单独 `@`。

预期：

- 立即显示候选列表。
- 候选列表中不包含当前用户自己。
- 候选列表只展示摘要。

### CASE-ROUTE-004 点击候选自动补全

步骤：

1. 输入单独 `@`。
2. 点击候选“角色1”。

预期：

- 输入框自动补全为 `@角色1` 或 `@角色1 `。
- 光标位于补全文本末尾。

### CASE-ROUTE-005 严格匹配

步骤：

1. 群组内存在角色“角色1”。
2. 发送 `@角色` 或 `@role1`。

预期：

- 不命中“角色1”。
- 返回明确错误提示或不触发路由。

### CASE-ROUTE-006 未 @ 任何角色

步骤：

1. 发送普通文本，不含任何 `@角色`。

预期：

- 前端或服务端阻止发送，或返回明确提示。
- 不应触发任何群角色处理。

### CASE-ROUTE-007 @ 不存在角色

步骤：

1. 发送 `@不存在角色 请处理`。

预期：

- 返回明确错误。
- 不触发 provider。

### CASE-ROUTE-008 同一角色重复 @

步骤：

1. 发送 `@角色1 @角色1 请处理`。

预期：

- 路由去重。
- 只触发一次。

## 5.4 角色管理

### CASE-MGMT-001 打开角色管理

步骤：

1. 进入群组页。
2. 点击“角色管理”。

预期：

- 弹窗打开。
- 展示当前群组角色列表。

### CASE-MGMT-002 移除角色成功

步骤：

1. 在角色管理中点击某角色“移除”。
2. 确认移除。

预期：

- 该角色从群组成员列表移除。
- 后续发送 `@该角色` 时应报错或提示不存在。
- 历史消息仍正常展示。

### CASE-MGMT-003 移除后的旧消息保留

前置条件：

- 角色1有历史回复记录。

步骤：

1. 移除角色1。
2. 刷新页面或重新载入群组。

预期：

- 历史中仍能看到角色1以前的消息。
- 仅成员列表中不再展示。

## 5.5 回流与展示

### CASE-MSG-001 群角色回复展示发送者身份

步骤：

1. 发送 `@角色1` 消息并等待回复。

预期：

- 回复消息清晰显示发送者为角色1。
- 不混淆为默认助手。

### CASE-MSG-002 回复中包含 @我

步骤：

1. 模拟角色回复 `@我 已收到`。

预期：

- 文本内容完整展示。
- 发送者身份仍为角色，不是用户。

### CASE-MSG-003 被 @ 对象展示在气泡右下角

步骤：

1. 模拟角色1回复 `@我 已收到`。
2. 模拟角色2回复 `@角色1 我已处理完成`。

预期：

- 正文与被 `@` 对象信息分开展示。
- 气泡右下角展示被 `@` 对象名字。
- 若对象存在头像或 emoji，则同步展示。

## 5.6 回归测试

### CASE-REG-001 普通角色 panel 创建不受影响

步骤：

1. 在原始角色列表页创建普通角色。

预期：

- 现有流程不变。
- 仍创建顶层 panel。

### CASE-REG-002 普通 panel 发消息不受影响

步骤：

1. 在普通角色窗口发送消息。

预期：

- 仍按当前逻辑投递给 provider。
- SSE 与消息展示无回归。

### CASE-REG-003 登录、登出、页面刷新不受影响

预期：

- 不影响已有认证流程和仪表盘初始化逻辑。

## 6. 建议新增自动化测试点

优先级 P0：

- `@角色` 解析与命中测试
- 群组角色创建/移除 store 测试
- 群消息发送 API 测试
- 回流消息正确映射为群角色消息

优先级 P1：

- 群组页面 server data 聚合测试
- 角色管理弹窗组件交互测试
- 创建群组/添加角色表单校验测试

## 7. 风险点

- 现有 `panel` 是顶层核心对象，群组引入后若复用不当，容易导致普通 panel 与群组消息模型混淆。
- provider 现阶段按单 `panel` / 单 `agentId` 投递，群消息若要多角色分发，需要新增路由层，否则会破坏现有链路。
- JSON 单文件存储在群组、角色、消息关系扩展后，读写复杂度会明显上升。

## 8. 上线前检查清单

- 新旧数据格式兼容。
- 老用户打开首页不会因缺少群组字段报错。
- 未配置群组数据时，UI 可正常回退。
- 群组与普通角色的创建、删除、发消息均已人工回归。
